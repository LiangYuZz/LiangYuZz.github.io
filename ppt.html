RuleEyes Relay Runtime · 技术架构介绍

面向 Android App 的自治审计运行时

用 Relay + ADB + Frida 把页面观察、动作规划、运行证据、失败重放串成一个可复盘的自动化审计闭环

Node.js / TypeScript ADB Frida Anthropic / OpenAI Relay SSE Web UI

当前版本聚焦：Android 动态审计、自治运行、证据链留存、失败重放

项目定位：不是单点工具，而是自治运行时

为什么传统方式不够

  • •静态扫描看得到声明和代码，但看不到真实页面操作后触发的运行时行为。
  • •人工测试能发现问题，但成本高、覆盖不稳定、结果难复现。
  • •单独用大模型会“看图会说”，但不一定能稳定驱动真实安卓设备完成闭环。
  • •单独用 Hook 也不够，因为看不到“它是在什么页面、什么操作后发生的”。

RuleEyes 的工程解法

  • ✓本地观察：截图、UI dump、manifest、Frida 事件一起采集。
  • ✓远端规划：把多模态观察交给 Relay，让模型返回下一步结构化动作。
  • ✓本地执行：由 ADB / Frida 执行动作，而不是把执行权交给云端黑盒。
  • ✓证据落盘：请求、响应、截图、事件、诊断文件统一保存，可追责、可回放。

项目核心难点

难点不在“单点能力”，而在于把 Relay 规划、ADB/截图/UI dump、Frida 证据 串成可复现的自治闭环。

多源观测要对齐

ObservationService + FridaEvidenceService

  • • 截图、UI dump、前台 Activity、Frida 事件来自不同通道，时间粒度和稳定性不一致。
  • • 单靠截图看不见后台敏感调用，单靠 Hook 又不知道调用发生在哪个页面。
  • • 必须把“页面状态 + 用户动作 + 后台行为”绑定成同一步证据链。

规划必须可执行

PlannerService + ExecutionService

  • • 大模型擅长理解页面，但不天然保证输出动作一定能在安卓设备上稳定执行。
  • • 需要把 Relay 返回约束成结构化 toolCalls，并经本地校验器过滤后再执行。
  • • 还要防止“过早结束”、误判已覆盖、以及重复无效动作循环。

外部 Relay 不稳定

RelayProvider + relay:replay

  • • 真实运行依赖第三方 API Relay，可能出现 upstream error、流式中断、额度异常。
  • • 健康检查通过，不代表“带截图、长上下文、流式输出”的真实规划请求一定成功。
  • • 所以系统必须保存 raw request / response，并支持失败重放与诊断。

闭环结果必须可复盘

TranscriptService + Web UI + SSE

  • • 这不是一次性 demo，而是要让每一步为什么点、点后看到什么、后台发生什么都能回放。
  • • 因此需要 transcript、诊断文件、截图历史、实时事件流统一落盘并前端可视化。
  • • 只有这样，结果才能用于审计、自证和后续整改闭环。

Step 1

  • • UI dump
  • • top activity
  • • manifest

Step 2

Plan

把观察结果发给 Relay，让模型返回结构化 toolCalls，而不是自然语言建议。

  • • 多轮 history 进入上下文
  • • 图片一并送入模型
  • • 约束输出为 JSON toolCalls
  • • 支持流式事件回传前端

Execute

本地校验并执行动作，再重新观察，进入下一轮，直到满足覆盖要求或失败退出。

  • • tap / swipe / input / keyevent
  • • Frida session 管理
  • • after-action observation
  • • transcript / raw payload 落盘

关键模块

工程拆分方式

项目不是把所有能力堆进一个 Agent，而是明确分成 本地执行层、远端规划层、证据与诊断层。

本地可控

所有设备动作与 Frida 行为都由本地服务执行，不依赖云端黑盒自动化。

模型只负责规划

模型负责理解和下一步决策，执行权限仍然收口在本地动作集合中。

诊断独立可复查

请求、响应、结构化输出、截图和 UTF-8 检查都能单独落盘，不依赖终端肉眼判断。
